# Normal Task Skill Index

本文件是 `normal` 的执行参考。

当当前 task 已经确定为 `normal` 时，normal executor 应结合：
- 当前 `task_content`
- recent `tasks`
- 本 skill index

来决定回复模式与表达边界。

---

## 核心原则

1. `reply` 面向用户
2. `task_result` 面向系统记录
3. 只能基于已有上下文作答
4. 缺少关键事实时，明确指出缺失项
5. 不讨论内部 controller 逻辑

---

## 一、结果解释

适用情况：

- 用户询问某次测试结果是什么意思
- 用户追问失败原因
- 用户想知道某个指标代表什么

回复目标：

- 直接解释结果
- 说明关键失败点或关键指标
- 不扩展成新任务

推荐表达：

- 最近一次测试的核心结论是……
- 失败主要集中在……
- 这个结果说明……

---

## 二、历史总结

适用情况：

- 用户要求总结最近几次输出
- 用户要求提炼关键信息
- 用户要求整理历史任务结果

回复目标：

- 压缩历史结果
- 提炼核心信息
- 保持事实化

推荐表达：

- 最近几次任务的核心信息可以概括为……
- 历史结果主要体现为……
- 当前已知结论包括……

---

## 三、查阅类回答

适用情况：

- 用户要求看最近一次报告
- 用户要求查看某次任务输出
- 用户要求基于已有结果直接回答

回复目标：

- 只基于已有内容作答
- 不自行扩展不存在的细节

推荐表达：

- 目前已有记录显示……
- 从最近一次任务结果看……
- 当前上下文中可以确认的是……

---

## 四、使用指导

适用情况：

- 用户询问怎么用
- 用户询问下一步做什么
- 用户询问某类任务应该怎么准备

回复目标：

- 给出直接指导
- 保持简短、可执行
- 不虚构环境状态

推荐表达：

- 当前更合适的下一步是……
- 如果你要继续推进，建议先补充……
- 这个场景下通常先做……

---

## `task_status` 判定

### `done`
已有上下文足以完成当前 normal task。

### `failed`
缺少关键事实，无法可靠完成当前 normal task。

## `task_result` 写法

保持一句话、事实化、短摘要。

推荐：

- 已基于历史结果完成解释
- 已整理最近一次任务结果摘要
- 因缺少关键事实未完成 normal task
